# 技术决策

## 2026-04-14：将当前目录视为发布快照而不是源码仓库

### 原因

- 根目录不存在项目级源码入口或构建配置。
- 目录主体由可执行文件、运行时资源、`app.asar` 及解包原生模块组成。

### 影响

- 后续操作以归档、版本化、解包分析、上传托管为主。
- 不应假设可以直接执行常规源码项目的安装、构建、测试流程。

## 2026-04-14：先初始化本地 Git，再决定远程上传策略

### 原因

- 用户目标是上传到 GitHub。
- 在确认目标仓库前，先建立本地 Git 历史有助于后续提交、差异比对和文档沉淀。

### 影响

- 当前目录已执行 `git init -b main`。
- 远程仓库信息、可见性和最终上传策略仍需单独确认。

## 2026-04-14：拒绝直接原样推送整个目录到普通 GitHub 仓库

### 原因

- 已确认存在多个超过 GitHub 普通 Git 单文件限制的产物。
- 当前环境未安装 `git-lfs`。

### 影响

- 后续必须在以下路径中二选一或组合使用：
  - 安装并使用 Git LFS
  - 先分卷打包再上传
  - 只上传可读源码或解包后的轻量内容

## 2026-04-14：采用“公开仓库 + Release 完整快照”策略

### 原因

- 用户要求把整个目录上传到 GitHub，并作为 Release 保存。
- 当前目录存在多个超大文件，不适合直接进入普通 Git 历史。
- 当前环境可以稳定完成仓库创建、默认分支推送和 Release 资产上传。

### 影响

- 仓库 `dsxx134/moyaclaw` 的 `main` 分支仅保存说明文档与工程记忆文件。
- 完整目录以 `v7.6.13-full-2026-04-14` Release 资产形式保存。
- 仓库当前可见性为公开。
- 后续新增版本时，应继续沿用“分支放说明、Release 放完整产物”的模式。

## 2026-04-14：为 app.asar 单独提供压缩分卷下载

### 原因

- `app.asar` 原始体积接近 `500 MB`，直接下载成功率不理想。
- 实测压缩后可降到约 `210 MB`，适合继续切成多个更小的分卷。

### 影响

- 当前 Release 额外提供 `app.asar.zip` 与 3 个分卷文件。
- 仓库内新增：
  - `scripts/build-app-asar-assets.ps1`
  - `scripts/merge-app-asar-parts.ps1`
  - `DOWNLOAD_APP_ASAR.md`
- 后续如果 `app.asar` 仍需单独分发，默认继续沿用该方案。
